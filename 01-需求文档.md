# Windows桌面自动化平台 —— 需求文档

## 1. 项目背景

市面上已有按键精灵、影刀RPA、UiBot等桌面自动化工具，但存在以下问题：
- 要么"纯脚本"（学习成本高），要么"纯拖拽"（灵活性受限），二者不能自由切换
- 能力固化，用户无法自行扩展新的"积木"
- OCR、颜色识别等视觉能力与流程控制割裂，集成度低

本项目旨在打造一个**开放的、积木化的桌面自动化平台**：用户既可以用可视化拖拽搭建流程，也可以直接编辑JSON代码，两种方式操作的是同一份数据，随时互相切换、双向同步。平台本身只提供"基础能力容器"，具体能力（点击、找色、OCR、循环……）都是可插拔的"积木"，方便后续持续扩展。

## 2. 产品定位

- **形态**：Windows桌面客户端程序，带GUI，界面用Web技术（HTML/CSS/JS）实现
- **核心心智模型**："积木" = Block（一个原子能力节点），"流程" = 一棵由Block组成的树/图，"JSON" 与 "画布拖拽" 是同一份流程数据的两种视图
- **目标用户**：需要做重复性电脑操作自动化的个人用户、想通过脚本精确控制流程的技术用户、希望通过拖拽零代码搭建流程的普通用户

## 3. 用户角色与典型场景

### 3.1 角色

| 角色 | 特点 | 使用方式偏好 |
|---|---|---|
| 零代码用户 | 不懂编程，希望通过点选/拖拽完成需求 | 纯界面拖拽模式 |
| 技术用户/开发者 | 熟悉JSON/编程逻辑，追求效率和精确控制 | 直接编辑JSON，或混合使用 |
| 平台扩展者（未来） | 希望给平台开发新Block（新能力插件） | 使用Block SDK/注册规范 |

### 3.2 典型使用场景

1. **重复性点击操作自动化**：用户录制一遍手动点击流程，程序自动循环回放（如批量处理某系统里的表单）
2. **状态判断类自动化**：判断屏幕某区域颜色是否变化（如游戏内血条、系统提示灯），据此决定下一步操作
3. **文字识别驱动流程**：识别屏幕某区域的文字（如网页状态、弹窗提示），根据识别结果走不同分支
4. **定时批量任务**：程序在设定时间/周期自动触发某个流程，无需人工值守
5. **混合编程**：用户先拖拽搭建大框架，再手动编辑JSON微调某个节点的参数（如精确坐标、容差值）

## 4. 功能性需求

### 4.1 基础能力（Block能力层）

| 编号 | 功能 | 说明 |
|---|---|---|
| F-01 | 鼠标点击/移动模拟 | 左键/右键/双击/拖拽，支持绝对坐标和相对坐标 |
| F-02 | 键盘输入模拟 | 单键、组合键、文本输入 |
| F-03 | 区域截图 | 指定屏幕矩形区域截图，供后续识别使用 |
| F-04 | 区域颜色识别 | 取指定坐标/区域的像素颜色，支持颜色容差比对 |
| F-05 | OCR文字识别 | 识别指定区域内文字，支持中英文，返回文字+置信度+位置 |
| F-06 | 图像模板匹配（找图） | 在屏幕或指定区域内查找目标小图，返回坐标 |
| F-07 | 点击流程录制 | 录制用户实际鼠标/键盘操作序列，转换为可编辑的节点序列 |
| F-08 | 流程回放 | 按节点顺序执行录制/编排好的流程 |
| F-09 | 延时/等待 | 固定延时，或"等待直到满足条件"（如等待某文字出现） |

### 4.2 编排能力（流程控制层）

| 编号 | 功能 | 说明 |
|---|---|---|
| F-10 | 条件分支（if/else） | 依据颜色识别/OCR结果/变量值等条件走不同分支 |
| F-11 | 多分支（switch/case） | 一个条件对应多个可能分支 |
| F-12 | 循环（固定次数） | 重复执行N次 |
| F-13 | 循环（条件循环） | while型循环，满足条件才继续/退出 |
| F-14 | 无限循环+退出条件 | 常驻监控型任务，需支持人工/条件终止 |
| F-15 | 定时触发 | 指定时间点或周期性（每N分钟/每天固定时刻）触发流程 |
| F-16 | 变量系统 | 节点输出可存为变量，后续节点可引用（如 `{{node_1.text}}`） |
| F-17 | 子流程/流程复用 | 一个流程可作为另一个流程中的节点调用（模块化） |

### 4.3 编程与编辑能力

| 编号 | 功能 | 说明 |
|---|---|---|
| F-18 | 可视化拖拽编排 | 画布形式，节点+连线，支持增删改连接 |
| F-19 | JSON代码编辑 | 提供代码编辑器（语法高亮+校验），可直接编写/修改流程JSON |
| F-20 | 双模式双向同步 | 拖拽操作实时反映到JSON，JSON编辑（校验通过后）实时反映到画布 |
| F-21 | Block参数表单化编辑 | 点选画布上的节点，右侧/弹窗显示该Block的参数表单 |
| F-22 | JSON Schema校验 | 用户手改JSON时，校验字段合法性，报错定位到具体节点 |

### 4.4 平台化/可扩展性

| 编号 | 功能 | 说明 |
|---|---|---|
| F-23 | Block注册表机制 | 新能力只需注册一份Schema+Handler，即可自动出现在界面积木面板中 |
| F-24 | Block分类展示 | 积木面板按"输入类""识别类""控制类""动作类"等分类展示，方便查找 |
| F-25 | 自定义Block（进阶，非MVP） | 允许技术用户编写Python脚本作为自定义Block接入平台 |

### 4.5 项目管理与运行控制

| 编号 | 功能 | 说明 |
|---|---|---|
| F-26 | 项目保存/加载 | 流程以JSON文件形式保存到本地，可重新打开编辑 |
| F-27 | 导入/导出 | 流程文件可导出分享，导入他人流程 |
| F-28 | 运行控制 | 开始/暂停/停止/单步执行 |
| F-29 | 执行日志 | 统一日志模型（system / runtime / audit / diag）：右侧面板默认可读运行日志并可展开 detail；系统与操作可过滤查看；诊断默认关闭；运行/应用日志可分别导出 |
| F-30 | 断点调试（进阶） | 可在某节点设断点，方便调试复杂流程 |

## 5. 非功能性需求

| 类别 | 需求 |
|---|---|
| 性能 | 单次截图+识别响应应在可接受范围内（截图<50ms，OCR识别<1s，取色<10ms），循环任务需支持长时间稳定运行不内存泄露 |
| 兼容性 | 支持 Windows 10/11，需考虑高DPI缩放对坐标定位的影响 |
| 可扩展性 | 新增Block不需改动核心执行引擎和前端框架代码 |
| 易用性 | 零代码用户能在不看文档情况下完成简单点击循环流程搭建 |
| 稳定性 | 长时间循环/定时任务不应导致程序崩溃或响应无关操作 |
| 数据安全 | 本地运行，流程数据默认存储在本地，不强制上传云端 |
| 可维护性 | 前后端通信协议、Block Schema需有清晰文档，便于后续维护和第三方扩展 |

## 6. 约束与假设

- 仅支持 Windows 桌面环境（不考虑 macOS/Linux）
- OCR/找图等视觉类能力依赖屏幕实际渲染内容，不支持识别被遮挡/最小化窗口内的内容（除非采用离屏渲染方案，作为进阶需求）
- 首个版本（MVP）不含"自定义Block脚本"能力，Block集合由平台内置提供
- 高DPI缩放、多显示器场景需要在设计阶段就考虑，避免后期大改

## 7. MVP范围界定

**第一期（MVP）必须包含：**
F-01～F-04, F-07～F-14, F-16, F-18～F-23, F-26, F-28～F-29

**第二期扩展（多数已落地）：**
F-05（OCR）、F-06（找图）、F-15（定时，含任务落盘与启动恢复）、F-17（子流程）

**已落地（原第三期项）：**
F-25（自定义 Block / `python_script` + `user_blocks`）、F-30（断点调试 + 变量监视）、窗口管理积木、相对窗口坐标录制、流程 `.flow.zip` 资源打包、多显示器虚拟桌面坐标与按监视器 DPI、插件模式（置顶半透明浮层 + 可调透明度 / 点击穿透，推荐无边框全屏）

**远期：**
离屏/最小化窗口 OCR（PrintWindow 等）

## 8. 验收标准（示例）

- 用户能够通过录制功能，录制一段"打开某程序→点击某按钮→输入文字"的操作，回放后与录制时行为一致
- 用户能够搭建一个"识别区域颜色→若为红色则点击A，否则点击B"的流程，并成功执行
- 用户在画布上拖拽增加一个循环节点后，切换到JSON编辑器可以看到对应的JSON结构已自动更新
- 用户手动编辑JSON中某节点的坐标参数并保存，切换回画布模式后节点参数正确更新，无需刷新整个画布
