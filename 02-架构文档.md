# Windows桌面自动化平台 —— 架构文档

## 1. 设计目标

1. **JSON与拖拽同源**：画布和代码编辑器是同一份流程数据的两种视图，不允许出现"两套逻辑各自维护"的情况
2. **能力可插拔**：新增一个自动化能力（Block），只需要"写一个Handler + 注册一份Schema"，不需要改动执行引擎或前端框架代码
3. **前后端分离但强耦合于同一进程**：界面用Web技术写，但通过桌面容器（而非独立浏览器）与本地Python能力层通信，保证能操作系统级API（鼠标/键盘/截图）

## 3. 技术选型

| 层级 | 技术 | 选型理由 |
|---|---|---|
| 桌面容器 | **pywebview** | 内嵌系统WebView2，体积小、支持JS↔Python双向调用，比Electron轻量得多 |
| 画布编辑器 | **React Flow** | 成熟的节点连线画布库，纯前端，二次开发接入自有节点类型和数据结构成本低，React生态下的事实标准 |
| 代码编辑器 | **Monaco Editor**（`@monaco-editor/react`） | VSCode同款，支持JSON语法高亮、错误提示、格式化 |
| 前端框架 | React 18 | 生态成熟、与React Flow同源同生态，组件化开发效率高 |
| 全局状态管理 | Zustand（或 Redux Toolkit） | 管理FlowModel这类跨组件共享的核心状态，Zustand更轻量，样板代码少 |
| 执行引擎 | Python | 与所有Capability库同语言，避免跨语言桥接损耗 |
| 鼠标键盘模拟 | pyautogui / pynput | pynput更适合底层监听（录制），pyautogui更适合高层动作模拟 |
| 截图 | mss | 截图性能远高于PIL自带方案 |
| 取色 | Pillow | 配合mss截图结果做像素读取 |
| OCR | RapidOCR（或PaddleOCR） | 中文识别效果优于Tesseract，且可离线部署 |
| 找图 | OpenCV（模板匹配） | 成熟稳定，支持多尺度匹配 |
| 定时调度 | APScheduler | 支持interval/cron/date多种触发方式 |
| 打包 | PyInstaller / Nuitka | 生成独立exe，无需用户装Python环境 |

## 4. 核心模块设计

### 4.1 FlowModel（流程数据模型）—— 整个系统的"单一数据源"

```json
{
  "flow_id": "flow_xxx",
  "name": "示例流程",
  "variables": {},
  "nodes": {
    "node_001": {
      "type": "click",
      "params": { "x": 100, "y": 200, "button": "left" },
      "next": "node_002"
    },
    "node_002": {
      "type": "if_color_match",
      "params": { "region": [0,0,100,100], "color": "#FF0000", "tolerance": 10 },
      "then": "node_003",
      "else": "node_004"
    },
    "node_003": { "type": "loop_n", "params": { "times": 5 }, "body": "node_005", "next": "node_006" }
  },
  "entry": "node_001"
}
```

**要点：**
- 所有节点存在一个扁平的 `nodes` 字典里（而非嵌套树），通过 `next`/`then`/`else`/`body` 等字段建立指向关系。这样画布连线（图结构）和执行顺序（有向图遍历）天然一致，也方便JSON文本编辑时定位单个节点
- 前端画布、JSON编辑器、后端执行引擎，三者都只认这一份数据结构，互相之间通过它同步，不允许任何一方维护"私有状态"

### 4.2 Block注册表（能力可插拔的核心）

每个Block由一份Schema描述（前端用来生成UI）和一个Handler函数（后端实际执行逻辑）组成：

```python
# blocks/ocr_recognize.py
SCHEMA = {
    "type": "ocr_recognize",
    "label": "OCR文字识别",
    "category": "识别类",
    "inputs": [
        {"name": "region", "type": "rect", "label": "识别区域"},
        {"name": "lang", "type": "select", "options": ["中文", "英文"], "default": "中文"}
    ],
    "outputs": [
        {"name": "text", "type": "string"},
        {"name": "confidence", "type": "number"}
    ]
}

def handler(params, context):
    region = params["region"]
    text, conf = run_ocr(region, params.get("lang", "中文"))
    return {"text": text, "confidence": conf}
```

启动时，后端扫描 `blocks/` 目录自动注册所有Block，并把所有Schema通过Bridge传给前端：

```python
BLOCK_REGISTRY = {}

def register_all_blocks():
    for module in scan_blocks_dir():
        BLOCK_REGISTRY[module.SCHEMA["type"]] = {
            "schema": module.SCHEMA,
            "handler": module.handler
        }
```

前端拿到Schema列表后，**自动生成积木面板的分类列表 + 每个Block的参数编辑表单**，不需要为每个新Block单独写前端代码——这是实现"平台化、随意积木搭配"的关键机制。

### 4.3 执行引擎（Interpreter）

```python
def run_flow(flow, start_node_id, context=None):
    context = context or {}
    node_id = start_node_id
    while node_id:
        node = flow["nodes"][node_id]
        block = BLOCK_REGISTRY[node["type"]]
        resolved_params = resolve_variables(node["params"], context)  # 替换 {{node_x.y}} 引用
        result = block["handler"](resolved_params, context)

        # 写入变量上下文，供后续节点引用
        for out_name, val in (result or {}).items():
            context[f"{node_id}.{out_name}"] = val

        node_id = decide_next(node, result, context)  # 处理 next/then/else/loop等跳转逻辑
    return context
```

- **条件节点**（if/switch）：Handler返回布尔值或分支标识，`decide_next` 据此选择 `then`/`else`/`case_x` 对应的 `node_id`
- **循环节点**：`decide_next` 内部维护循环计数或条件判断，未结束则跳回 `body` 的入口节点，结束后跳到 `next`
- **变量替换**：所有 `params` 在执行前统一走一次 `resolve_variables`，将形如 `"{{node_001.text}}"` 的字符串替换为 `context["node_001.text"]` 的实际值

### 4.4 前后端通信协议（JS-Bridge）

pywebview 暴露一个 Python API 类，前端通过 `window.pywebview.api.xxx()` 调用：

```python
class Api:
    def get_block_registry(self):
        return [b["schema"] for b in BLOCK_REGISTRY.values()]

    def run_flow(self, flow_json):
        return run_flow(json.loads(flow_json), entry_node)

    def start_recording(self):
        recorder.start()

    def stop_recording(self):
        return recorder.stop()  # 返回录制生成的节点序列

    def save_flow(self, flow_json, filepath):
        ...

    def load_flow(self, filepath):
        ...
```

执行过程中的实时日志/进度，通过 `window.evaluate_js()` 从Python主动推送到前端（因为pywebview的调用默认是同步请求-响应，长时间执行的流程需要用回调方式往前端推事件）。

### 4.5 双模式同步机制

```
用户拖拽画布节点
    ↓
React Flow 触发 onNodesChange/onConnect 回调
    ↓
更新 Zustand 中的全局 FlowModel（单一状态源）
    ↓
FlowModel 变化 → 序列化为格式化JSON → 更新 Monaco 编辑器内容（如果代码视图当前可见）

用户编辑 Monaco 中的JSON文本
    ↓
监听 onChange，防抖后触发解析
    ↓
JSON.parse + Schema校验（Ajv等JSON Schema校验库）
    ↓
校验通过 → 更新 Zustand 中的 FlowModel → React Flow 因订阅了该状态而自动重新渲染画布
校验失败 → 在Monaco中标红对应行，不更新FlowModel（避免脏数据传导到画布）
```

要点：**两个方向都不能"手写转换脚本互相同步"，而是都统一收敛到 Zustand 里的一份 FlowModel，画布和代码编辑器都只是它的订阅者/渲染组件。** 这样以后即使要加第三种视图（比如"自然语言描述视图"），也不需要再写一套新的双向转换逻辑，只需要再订阅同一个store即可。

## 6. 打包与部署

- 前端用 Vite 构建为静态资源（dist目录），由pywebview直接加载本地HTML文件（`webview.create_window(url='frontend/dist/index.html')`）
- 后端连同前端静态资源一起，用 PyInstaller 打包为单一exe
- RapidOCR等模型文件需要作为数据文件一并打包（注意PyInstaller的 `--add-data` 配置）
- 需要处理Windows下的管理员权限问题（模拟点击某些窗口可能需要以管理员身份运行才生效）

## 7. 可扩展性设计要点

1. **Block即插件**：`blocks/` 目录下新增一个 `.py` 文件并按规范导出 `SCHEMA` 和 `handler`，重启程序即可在积木面板看到新能力，无需改动前端和执行引擎代码
2. **节点类型可分层扩展**：未来如果要支持"子流程"节点，本质上是一种特殊Block，其Handler内部递归调用 `run_flow`，复用同一套执行引擎，不需要额外设计
3. **通信协议前向兼容**：Api类新增方法即可扩展前端可调用能力，不影响已有方法
4. **未来自定义Block（第三期）**：可设计为用户编写符合规范的Python脚本放入 `blocks/user/` 目录，注册机制天然支持热加载，无需额外改造

## 8. 关键技术风险与应对

| 风险 | 应对方案 |
|---|---|
| 高DPI缩放导致坐标偏移 | 启动时通过Windows API获取实际DPI缩放比例，所有坐标操作前统一做缩放换算 |
| pywebview长时间执行阻塞UI线程 | 执行引擎放到独立线程/进程运行，通过 `evaluate_js` 异步推送日志和状态，避免卡死界面 |
| OCR/找图性能瓶颈影响循环速度 | 提供"识别频率"配置项，避免在高频循环中每次都全量截图识别；模型加载做单例缓存，避免重复加载 |
| JSON手改导致运行时崩溃 | 执行前对整个FlowModel做一次Schema级校验，校验不过直接拒绝运行并定位报错节点，不允许"半合法"数据进入执行引擎 |
