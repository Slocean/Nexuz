# Nexuz 平台功能不足分析

更新日期：2026-07-19

本文基于当前代码、README 与 [`REMEDIATION_PLAN.md`](./REMEDIATION_PLAN.md)，梳理相对「可用的 Windows 桌面自动化 MVP」仍存在的功能与产品缺口，供后续排期参考。

执行原则与安全整改项仍以 `REMEDIATION_PLAN.md` 为准；本文侧重**用户可感知能力**与**交付完整性**，并交叉标注未闭环的 P0/P1 项。

---

## 1. 结论摘要

平台已具备可视化编排、积木执行、调试、录制、导入导出与热更新等主路径，但仍是 MVP：

| 维度 | 状态 |
| ---- | ---- |
| 画布编排 / 运行控制 | 可用 |
| 常用桌面自动化积木 | 可用 |
| Try / Catch / 失败恢复 | 已落地 try_catch（body/catch/finally）；节点级 on_error 重试策略仍缺 |
| 脚本与用户积木隔离 | 部分完成，未到目标态 |
| 第三方流程信任与权限 | 偏薄 |
| Flow AI | 生产隐藏，未接入 |
| 停止与资源回收 | 不完整 |
| 错误反馈与工程可维护性 | 待收口 |

优先顺序建议：先收完隔离尾项与停止/信任/错误反馈，再恢复 Flow AI 与扩展积木能力。

---

## 2. 未闭环的核心能力

### 2.1 脚本 / 插件隔离未到目标态

对应整改：`P0-S1`、`P0-P1`（阶段二）。

已完成：

- Python 脚本迁入独立 worker，主进程不再直接 `exec`
- Job Object 限制内存、子进程数、最长运行时间
- Windows worker 降为 Low integrity；常规网络、子进程、文件写入默认阻断
- 用户积木静态解析 `SCHEMA`、授权哈希、一次性插件 worker

仍不足：

- 未授权文件读取、原生扩展绕过仍待 AppContainer / 独立低权限账户收口
- 不得将进程隔离或语言级过滤宣传为「安全沙箱」；对外仍须标明「仅可信代码」

### 2.2 Flow AI 未上线

对应整改：`P1-A1`、阶段五第 4 项。

- 生产构建已隐藏不可用入口；仅 Vite 开发预览保留
- 恢复前必须：改走 Python Bridge、配置后端服务、接入正式 `addNodeFromSchema`
- 设计稿中的 AI 控件（如 Flow AI Helper、原 AI 积木卡片）尚未接入真实链路

### 2.3 停止语义不完整

对应整改：`P1-STOP`（阶段四）。

缺口：

- 缺少统一的子进程 / Frida 会话 / worker registry
- 部分阻塞能力未实现可靠取消
- 停止后未保证等待资源回收；超时后终止进程树的路径未全面覆盖

风险：用户点击「停止」后，仍可能存在继续运行的旧命令、RPC 或 worker。

### 2.4 信任与权限模型偏薄

对应整改：`P1-TRUST`、`P1-BRIDGE`（阶段四）。

已有外部流程「先解析、能力清单、显式确认」的基础，仍缺：

- 受限模式
- 授权目录
- 系统积木策略
- 审计日志
- Bridge 按 flow / window / update / frida / filesystem 拆分的最小 wrapper
- 清库、更新、命令、Frida 等的短期 capability token 与原生确认
- 生产模式不暴露开发 API

结果：第三方流程尚不能默认「最小权限可跑」；前端 XSS / 调试注入面偏大。

### 2.5 Try / Catch（流程级异常处理）

已落地（2026-07-19）：

- 控制积木 `try_catch`：出口 `body` / `catch` / `finally` / `next`
- 解释器将异常路由到最近一层仍处于 body 的 try；输出 `raised` / `error`
- 无 catch 仅有 finally 时，收尾后重新抛出异常

仍缺：

- 节点级 `on_error = abort | continue | goto` 与有限次重试
- 画布 try 体范围框（循环体框已有，try 尚未对齐）

## 3. 产品与体验缺口

### 3.1 错误反馈不统一

对应整改：`P1-ERROR`。

- toast / dialog 未统一
- 保存、导入、导出、运行失败有时只写日志，用户不可见
- 校验失败缺少「自动选中首个问题节点并聚焦字段」

### 3.2 可维护性拖累功能迭代

对应整改：阶段五。

- `backend/api.py` 仍偏巨型，待按领域拆分并保留兼容门面
- Bridge 客户端与 store slices 跨域订阅偏多
- Inspector 参数编辑器未按输入类型拆成独立组件

影响：后续加积木、改权限、修停止语义的改动面大、回归成本高。

### 3.3 发布与热更新验收未完全闭环

对应整改：阶段一「真实发布验收」。

代码侧门禁（签名强制、SHA-256、证书锚、CI/Release）已落地，仍待：

1. 配置签名秘密后打真实签名包
2. 确认 Release 含 `Nexuz.exe` 与 `Nexuz.exe.sha256`
3. 用上一版受信任构建走通下载 → 校验 → 替换 → 重启
4. 分别篡改 exe、sha256、下载 URL、签名证书，确认四类均在替换前失败

另：README 中部分「热更新不依赖签名」的表述与现整改方向不一致，文档需随后对齐。

## 4. 相对成熟 RPA / 自动化平台的能力边界

以下并非当前整改必做项，但构成产品定位上的不足：

| 能力 | 现状 |
| ---- | ---- |
| 识别方式 | 偏坐标、图像、OCR、窗口标题；缺成熟 UI 控件树 / 无障碍树 |
| 协作与分发 | 本地 `.flow.json` / `.flow.zip`；缺云同步、流程中心、权限协作 |
| 运行形态 | 本机桌面为主；缺远程 agent、无人值守调度中心 |
| 密钥与账号 | 无系统级凭据保险库 |
| 跨平台 | 仅 Windows（pywebview / WebView2） |
| 调度可靠性 | 有 `schedule_trigger` 与定时落盘；缺企业级调度、失败重试策略、告警链路 |
| 智能编排 | Flow AI 未接入 |

## 5. 建议排期（与整改计划对齐）

| 优先级 | 主题 | 主要验收 |
| ------ | ---- | -------- |
| P0 | 脚本隔离收口（AppContainer / 低权限账户） | 未授权读盘与原生扩展绕过被阻断；逃逸仅限 worker |
| P0 | 真实签名包端到端验收 | 四种篡改场景均失败；正常更新成功 |
| P1 | Bridge 拆分 + capability token | 未授权高危 API 不可直接调用 |
| P1 | 受限模式 / 授权目录 / 审计 | 第三方流程默认受限 |
| P1 | 统一错误反馈 | 关键失败可见；校验失败聚焦首个问题节点 |
| P1 | 停止语义与资源回收 | 停止后无残留命令 / RPC / worker |
| P1 | 节点 on_error / 重试策略 | 单节点失败可继续或跳转，无需包一层 try_catch |
| P1 | api / Bridge / Inspector 拆分 | 领域测试可覆盖核心改动 |
| 其后 | 恢复 Flow AI | Bridge + 后端服务 + `addNodeFromSchema` |
| 其后 | 产品能力扩展 | 控件树、调度、凭据等按产品目标单独立项 |
